# 프로젝트 제목

**Impact Map: 바이브 코딩 환경을 위한 변경 영향도 자동 분석 프레임워크**

---

# 문제 정의

### 1) 바이브 코딩 환경에서 발생하는 문제

LLM 기반 바이브 코딩 환경에서는 코드 작성과 수정 속도가 비약적으로 증가하지만, 그에 비례해 **변경 코드의 영향 범위를 정확히 인지하지 못한 채 배포되는 문제**가 빈번하게 발생한다.

개발자는 자연어 기반 지시를 통해 다수의 파일과 로직을 동시에 수정하며, 이 과정에서 함수 호출 관계, 데이터 흐름, 설정 의존성, 외부 인터페이스 변화에 대한 인지 없이 코드가 생성·수정된다.

그 결과, 변경 사항이 실제로 **어떤 API, 어떤 서비스, 어떤 데이터, 어떤 운영 설정에 영향을 미치는지**를 개발자 개인의 경험이나 PR 리뷰에 의존하게 되며, 이는 대규모 시스템일수록 위험도를 급격히 증가시킨다.

### 2) 관련되는 품질·안정성·보안·유지보수 측면

- **품질(Quality)**: 변경 영향이 누락되어 테스트 범위가 부정확해지고, 회귀 버그 발생 가능성 증가
- **안정성(Stability)**: 예상치 못한 호출 체인 붕괴, 런타임 장애, 배포 후 롤백 증가
- **보안(Security)**: 권한·설정·외부 인터페이스 변경에 따른 보안 취약점 사전 인지 실패
- **유지보수(Maintainability)**: 시스템 구조와 의존성에 대한 문서 부재로 기술 부채 가속

즉, 바이브 코딩은 **속도를 얻는 대신, 시스템 전체에 대한 통제력을 잃는 구조적 문제**를 내포하고 있다.

---

# 문제의 원인 분석

본 문제의 근본 원인은 다음 세 가지로 정리할 수 있다.

### 1) 변경 중심 개발 방식의 한계

기존 개발 프로세스는 “코드 변경 → 영향 추론 → 리뷰”의 흐름을 사람 중심으로 설계해왔다. 그러나 바이브 코딩 환경에서는 변경 단위가 커지고 빈도가 높아져, **인간이 변경 영향도를 직관적으로 판단하기 어려운 규모**가 된다.

### 2) 아키텍처·의존성 정보의 비가시화

실제 시스템은 코드, 데이터베이스, 설정 파일, 클라우드 리소스, 외부 API 등 다층 구조로 이루어져 있으나,
이 관계를 **기계가 해석 가능한 형태로 정리한 아키텍처 맵이 부재**한 경우가 대부분이다.

### 3) 영향 분석의 비정형성

현재의 영향 분석은 체크리스트, 경험, 리뷰 코멘트 등 **비정형적·주관적 방식**에 의존하고 있으며,
정적 분석을 통해 “영향 가능 영역”을 기술적으로 산출하는 체계가 부족하다.

이로 인해, **빠른 코드 생성 환경에 대응할 수 있는 기술적 안전장치의 필요성**이 대두된다.

# 개선 아이디어 핵심 제안

본 제안은 **정적 분석 기반 변경 영향도 맵(Impact Map) 프레임워크**를 통해 문제를 해결한다.

### 핵심 아이디어

> “**변경된 코드(diff)**를 입력으로 받아,
> **해당 변경이 시스템 전반에 미칠 수 있는 영향 범위**를
> 호출 관계·데이터 흐름·설정 의존성·외부 인터페이스 관점에서 **자동 산출·가시화**한다.”
> 

### 2-Agent 구조 설계

### (Sub) 아키텍처 맵 작성 에이전트

- 코드 저장소, 인프라 정의(IaC), 설정 파일 등을 스캔
- 함수/모듈/서비스 호출 관계
- 데이터베이스·메시지 큐·스토리지 접근 관계
- 설정(env, secret, 권한) 의존성
    
    → **멀티 레이어 아키텍처 맵(그래프)** 자동 생성
    

### (Main) 변경 영향도 분석 에이전트

- PR 또는 커밋 diff 입력
- 변경된 코드 노드를 아키텍처 맵에 매핑
- 영향 전파 경로를 그래프 탐색 방식으로 계산
- 영향 범위를 **확정 / 가능 / 추정** 단계로 분류
- 리스크 점수 및 근거 경로(Evidence) 함께 제공

### 결과물

- 배포 전 한 장으로 보는 **Impact Summary 리포트**
- 변경 코드 중심의 **영향도 시각 맵(Impact Map)**
- 필수 테스트, 검토 대상, 운영 체크리스트 자동 제안

# 구현된 주요 기능

### 1) 아키텍처 의존성 자동 스캔

코드 저장소를 스캔하여 모듈(파일) 간 **의존성 참조 그래프를 자동 구축**한다.

- 전체 저장소 또는 특정 파일 단위로 의존성 스캔 가능
- AST(Abstract Syntax Tree) 구문 분석 또는 정규식 기반으로 함수·모듈 호출 관계를 추출
- 인접 리스트(Adjacency List) 기반 JSON 그래프로 출력하여, 이후 영향도 분석의 입력으로 사용
- 코드 변경 전, AI 에이전트가 수정 대상의 호출처·참조처를 사전에 파악하는 역할 수행

### 2) 변경 영향도 자동 분석 및 리포트 생성

PR 또는 커밋 Diff를 입력받아, 아키텍처 맵과 대조하여 **변경이 시스템에 미치는 영향 범위를 자동 산출**한다.

- 변경된 코드 노드를 아키텍처 그래프에 매핑하고, 그래프 탐색을 통해 영향 전파 경로를 계산
- 출력 리포트에는 다음이 포함됨:
  - 전체 위험도 등급 (🔴 High / 🟡 Medium / 🟢 Low)
  - 확정된 영향 범위 (Confirmed Blast Radius): API·서비스·인프라 계층별 분류
  - 에이전트 조치 권고 (Recommended Action): DB 마이그레이션, Fallback 코드, 리뷰어 승인 등

### 3) 영향도 시각화 (Web Viewer)

에이전트 분석 결과를 **인터랙티브 웹 UI 그래프로 시각화**하여 직관적으로 확인할 수 있다.

- 의존성 그래프를 네트워크 그래프 형태로 렌더링
- 변경 시뮬레이션 시 **파급 효과(Blast Radius)를 실시간으로 시각화**
  - 수정 대상 노드: 🔴 빨간색 강조
  - 직접 영향받는 노드: 🟠 주황색 강조
  - 연쇄 영향 경로: 빨간색 엣지로 표시
- 우측 패널에 **Impact Report 자동 표시**: 위험도, 파급 범위, AI 권고사항 포함

### 4) Zero-Prompt 자동화 (자동 방어 체계)

개발자가 별도 분석 명령을 내리지 않아도, **배포 전 자동 안전장치**가 작동한다.

#### AI 에이전트 자동 방어 (PreToolUse Hook)
- AI 에이전트가 커밋이나 PR 생성 명령을 실행하려 할 때 **시스템이 자동으로 인터셉트**
- 변경사항을 추출하여 영향도 분석을 백그라운드에서 실행
- 고위험(High Risk) 감지 시 **커밋을 자동으로 차단**하고 경고 메시지를 출력

#### Git 프리커밋 안전장치
- IDE나 터미널에서 직접 커밋 시에도 **자동 영향도 스캔**이 수행됨
- 스테이징된 변경사항을 분석하여 고위험 변경 시 **커밋 차단**
- 긴급 시 강제 커밋 가능하나, CI/CD 단계에서 재검증됨

### 5) CI/CD 파이프라인 자동 연동

PR 생성 시 **자동 영향도 분석 및 결과 코멘트**를 수행한다.

- PR이 올라오면 CI 파이프라인이 자동으로 아키텍처 맵 생성 → 영향도 분석 → 리포트 생성
- 분석 결과를 PR 코멘트로 자동 등록하여, 리뷰어가 별도 도구 없이 바로 확인 가능
- 고위험 변경사항일 경우 리뷰어의 상세 검토를 유도하는 구조

### 6) Cross-Agent 지원 (Tool-Agnostic 아키텍처)

특정 AI 도구에 종속되지 않는 **범용 워크플로우 체계**를 제공한다.

- 모든 AI 어시스턴트가 준수해야 하는 **3-Phase 공통 프로세스** 정의:
  - Phase 1: 아키텍처 파악 (Pre-Check) — 코드 수정 전 의존성 사전 확인
  - Phase 2: 코드 작성 및 수정 (Execution) — 파악된 영향 범위를 고려한 수정
  - Phase 3: 변경 영향도 자가 검증 (Validation) — 영향도 분석 실행 및 리포트 제출
- **Cursor, Claude Code, Antigravity** 등 주요 AI 에디터에서 동일한 안전 검증 프로세스가 적용됨
- 에이전트용 스킬(Skill) 정의를 통해, AI가 복잡한 변경 시 자동으로 분석 스크립트를 실행하고 리포트를 생성

# 적용 시나리오 및 기대 효과

### 적용 시나리오

1. 개발자가 바이브 코딩으로 기능 구현 및 수정
2. PR 생성 시 Impact Map 자동 실행
3. 변경 diff 기반 영향도 분석 수행
4. PR 화면에서 영향 요약 리포트 확인
5. 리스크 수준에 따라 테스트/승인/배포 전략 결정

CI/CD 파이프라인과 연계하여 **고위험 변경은 자동으로 검증 단계를 강화**할 수 있다.

### 다층 방어 체계 (Defense in Depth)

본 프로젝트는 **단일 검증 지점이 아닌 5단계 다층 방어 구조**를 구현하였다:

| 단계 | 방어 지점 | 메커니즘 | 동작 방식 |
|------|-----------|----------|-----------|
| 0층 | 코드 수정 직전 | PreToolUse Hook (Edit/Write) | AI가 파일 수정 시도 시 영향도 사전 분석 → 개발자 확인 |
| 1층 | AI 에디터 내부 | Skills/Rules 기반 사전 인지 | AI가 코드 수정 전 의존성 자동 스캔 |
| 2층 | 커밋 시점 | Claude Hook / Git Pre-commit | 고위험 변경 커밋 자동 차단 |
| 3층 | PR 시점 | GitHub Actions CI/CD | PR에 Impact Report 자동 코멘트 |
| 4층 | 리뷰 시점 | 시각화 UI + 리포트 | 리뷰어가 Blast Radius를 시각적 확인 |

### 기대 효과

- **장애 예방**: 배포 전 영향 범위 사전 인지로 장애 발생률 감소
- **리뷰 효율 향상**: 리뷰어가 "어디를 봐야 하는지" 명확히 인지
- **운영 안정성 강화**: 설정·권한·외부 인터페이스 변경 리스크 조기 발견
- **바이브 코딩 신뢰성 확보**: 속도는 유지하면서 안전성 보완
- **에이전트 간 상호 검증**: 코드 생성 AI와 검증 AI의 분리를 통한 구조적 안전성

Impact Map은 단순한 PR 보조 도구가 아니라, **에이전트 기반 개발 체계에서 '안전성을 강제하는 검증 인프라'로 작동할 수 있다.** 코드 생성 에이전트가 기능을 구현하더라도, 배포 이전에 Impact Map이 변경 영향 범위를 자동 분석하여 시스템 전반의 리스크를 구조적으로 평가한다. 이는 인간 검토에 의존하던 영향 분석을 **에이전트 간 상호 검증 구조(Agent-to-Agent Validation)**로 전환하는 시도이며, 자율 개발 환경에서도 통제 가능한 안정성을 확보하는 기반이 된다.

# 한계 및 확장 가능성

### 현재 데모 한계

- 현 단계에서는 **Mock 데이터 기반 시연**으로, 실제 AST 파싱이나 LLM 추론은 미구현
- 데모용 고정 의존성 그래프를 사용하며, 동적 코드 분석은 미포함

### 정적 분석의 본질적 한계

- 동적 로딩, 런타임 분기, Feature Flag 기반 로직은 정적 분석에 한계 존재
- 외부 SaaS의 실제 동작이나 실시간 상태는 완전한 추론 불가
- 언어·프레임워크별 분석 정확도 차이 발생 가능

### 확장 가능성

- **AST 기반 실제 코드 분석**: Tree-sitter 등을 활용한 Python/JS/TS 실제 의존성 추출
- **LLM 기반 의미 분석**: OpenAI API를 활용한 변경 코드의 맥락적 영향 추론
- 런타임 트레이싱, 로그 기반 분석과 결합한 하이브리드 영향도 분석
- 보안 영향 분석(권한 상승, 데이터 노출 가능성) 자동화
- 변경 이력 기반 **사고 예측 모델**로 고위험 패턴 사전 감지
- 조직별 영향 분석 규칙 학습을 통한 맞춤형 Impact Map 진화
- **다중 언어·프레임워크 확장**: Java, Go, Rust 등 추가 언어 지원
